# 의사결정 기록

이 문서는 워크스페이스 환경 구축 과정에서 논의되고 결정된 사항들을 시간순으로 기록합니다.

---

## 2026-04-11

### 결정 1: 워크스페이스 디렉토리 구조

**논의 배경**:
9개 리포지토리가 이미 클론되어 있는 상태에서, 이 상위 디렉토리를 어떻게 구성할지 논의.

**제안된 구조**:
```
hellobot/
├── CLAUDE.md              # 워크스페이스 레벨 Claude 지침
├── docs/
│   ├── architecture.md    # 전체 아키텍처
│   ├── api-contracts/     # 파트 간 API 계약
│   └── decisions/         # 의사결정 기록
├── scripts/
│   └── setup.sh           # 전체 리포 관리 스크립트
└── (9개 리포지토리)
```

**결정**: 제안대로 채택. 워크스페이스 루트에 CLAUDE.md를 두어 Claude Code가 전체 맥락을 자동으로 로드하도록 함.

**근거**: 
- CLAUDE.md는 Claude Code가 자동 로드하는 파일이므로, 리포 간 관계와 규칙을 여기에 기록하면 모든 작업에서 참조됨
- docs/는 코드가 아닌 문서를 분리하여 관리
- scripts/는 크로스 리포 유틸리티 용도

---

### 결정 2: 각 리포의 기술 스택 조사 및 문서화

**작업 내용**:
9개 리포를 탐색하여 기술 스택, 역할, 의존 관계를 파악하고 CLAUDE.md와 architecture.md에 기록.

**파악된 스택**:
- 서버: Node.js / Express / TypeORM / PostgreSQL
- 스튜디오 서버: Java / Spring Boot / MongoDB
- 프론트엔드: Next.js (신규), Angular 13 (레거시/스튜디오)
- 모바일: Kotlin/MVVM (Android), Swift/ReactorKit (iOS)
- 데이터: Python / Airflow / BigQuery

**결정**: architecture.md에 시스템 구성도, 컴포넌트 상세, 도메인 정보까지 포함하여 상세 문서화.

## 2026-04-12

### 결정 3: 컨텍스트 효율성 규칙

**논의 배경**:
사용자가 "리포지토리 전체는 너무 거대하기 때문에 특정 프로젝트를 수행할 때 꼭 필요한 데이터만 참조했으면 좋겠다"고 요청.

**결정**: CLAUDE.md에 컨텍스트 관리 규칙을 명시적으로 추가.

**핵심 규칙**:
1. 컨텍스트 로딩 순서 정의 (CLAUDE.md → 피쳐 문서 → 리포 CLAUDE.md → 타겟팅 검색)
2. 담당 외 리포 소스코드 탐색 금지
3. 광범위 탐색 금지 — 항상 Grep/Glob으로 타겟팅
4. 크로스 리포 정보는 피쳐 문서(api-spec.md 등)로 전달

**근거**: 9개 리포 전체 코드량이 방대하여, 무작정 탐색하면 컨텍스트 낭비. 에이전트가 자기 영역만 집중하도록 규칙 필요.

### 결정 4: 역할별 에이전트 구성

**논의 배경**:
사용자가 "요구사항을 구체화하고, 기획하고, 설계하고, 구현하고, 검토하는 다양한 역할이 필요하다"고 요청. 각 역할을 담당하는 에이전트를 구성하여 필요한 과업마다 적절한 담당자와 논의하며 구체화하는 형태를 원함.

**제안된 역할**:
1. `/analyze` — PM/기획자
2. `/design` — 기술 설계자
3. `/dev-server` — 서버 개발자
4. `/dev-ios` — iOS 개발자
5. `/dev-android` — Android 개발자
6. `/dev-web` — 웹 개발자
7. `/dev-studio` — 스튜디오 개발자
8. `/review` — 코드 리뷰어

**사용자 피드백**: 데이터 엔지니어 역할도 추가 요청.

**최종 결정**: 9개 에이전트 커맨드 확정.
- 위 8개 + `/dev-data` (데이터 엔지니어, common-data-airflow 담당) 추가

**근거**:
- 역할 분리로 각 에이전트가 자기 전문 영역에 집중
- 커맨드로 호출하면 해당 역할의 시각/규칙이 자동 적용
- 에이전트 간 커뮤니케이션은 피쳐 문서를 통해 수행 (직접 코드 참조 방지)

### 결정 5: 워크플로우 단계

**결정**: 4단계 워크플로우 확정.

```
/analyze (분석) → /design (설계) → /dev-* (구현) → /review (검토)
```

**핵심 설계**:
- 구현 단계에서 파트별 병렬 진행 가능 (단, 의존 관계에 따라 순차도 가능)
- 일반적 순서: 서버 먼저 → 클라이언트(iOS, Android, Web) → 데이터
- 간단한 작업은 단계 생략 가능 (버그 수정은 바로 `/dev-*`)

**근거**: hellobot-server에 이미 존재하는 피쳐 개발 프로세스(설계→개발→완료)를 서비스 전체로 확장. 앞단에 분석 단계, 뒷단에 크로스 리포 검토 단계를 추가.

### 결정 6: 피쳐 문서 체계 — 워크스페이스 vs 리포

**논의 배경**:
hellobot-server에는 이미 `docs/features/` 에 상세한 피쳐 문서 체계(backend-design.md, backend-guide.md 등)가 존재. 워크스페이스 레벨에도 피쳐 문서가 필요한데, 역할이 겹치면 안 됨.

**결정**: 두 레벨로 분리.

| 레벨 | 위치 | 내용 |
|------|------|------|
| 워크스페이스 | `docs/features/` | 요구사항, 전체 설계, API 계약, 파트별 과업 |
| 리포 | 예: `hellobot-server/docs/features/` | 해당 파트 구현 상세 |

**근거**: 
- 워크스페이스 문서는 "무엇을/왜"에 집중, 리포 문서는 "어떻게"에 집중
- api-spec.md가 서버↔클라이언트 간 계약서 역할 → 클라이언트 개발자가 서버 코드를 읽지 않아도 됨 (컨텍스트 효율성)

사용자가 "처음 보는 사람도 사용할 수 있도록 자세히 설명"을 요청.

- 상황별 사용 패턴 5가지 (서버만, 버그 수정, 기획만, 큰 피쳐, 자유 질문)

### 결정 9: 워크스페이스 관리자 에이전트 추가

**논의 배경**:
사용자가 "워크스페이스 레벨의 상태를 관리하는 역할이 필요하다"고 요청.

**근거**: 다른 에이전트들은 자기 파트에 집중하므로, 워크스페이스 전체의 문서 일관성을 유지할 전담 역할이 필요.

### 결정 10: 프로젝트 디렉토리 + 워크트리 기반 개발 구조

**논의 배경**:
동료들과 동시 개발, 리포별 배포 방식 차이, 같은 리포에서 2~3개 피쳐 동시 진행 등 실제 협업 환경의 제약을 논의. 워크트리를 활용한 격리 개발 방식을 사용자가 제안.

**결정**: 구조 전면 변경.

1. **`docs/features/` → `projects/`**: 기존 피쳐 문서 디렉토리를 최상위 `projects/`로 이동. `docs/`는 프로젝트와 무관한 상시 문서용으로 유지.

2. **프로젝트 = 문서 + 워크트리**: 각 프로젝트 디렉토리 안에 문서와 워크트리가 함께 존재.
   ```
   projects/20260412-share-result/
   ├── readme.md, status.md, tasks.md, design.md, api-spec.md
   └── worktrees/
       ├── hellobot-server/    ← git worktree
       └── hellobot_iOS/       ← git worktree
   ```

3. **원본 리포 고정**: 원본 리포는 메인 브랜치에 고정. 코드 참조/검색용. 수정은 워크트리에서만.

4. **피쳐 브랜치 네이밍**: `feat/{프로젝트명}`

5. **워크트리 운영**:
   - 프로젝트 시작 시 자동 생성하지 않음 — 필요시 사용자 확인 후 생성
   - 작업 중 추가 리포 수정 필요 시 그때 워크트리 추가
   - 완료 후 바로 삭제하지 않음 (후속 hotfix 가능) — 별도 정리 단계에서 제거

**근거**:
- 동시 피쳐 개발 시 브랜치 전환/stash 불필요
- 원본 보호로 실수 방지
- 프로젝트 단위로 문서+코드가 한 곳에
- 동료의 작업과 물리적으로 격리

### 결정 11: 리포별 Git/배포 규칙 조사

**작업 내용**: 8개 리포의 Git 브랜치 전략, 배포 방식, CI/CD 설정을 전수 조사.

**발견**: 메인 브랜치(master/main/develop), 배포 방식(자동CD/배포브랜치/수동), 피쳐 브랜치 접두사가 리포마다 다름.

**결정**: 임시로 `repo-git-deploy.md`에 정리. 추후 각 리포의 CLAUDE.md에 정식 문서화.

**근거**: 에이전트가 워크트리를 생성할 때 각 리포의 메인 브랜치를 정확히 알아야 함.

### 결정 12: 워크스페이스 자체를 Git으로 관리

**논의 배경**:
워크스페이스 설정/문서를 버전 관리하고 싶지만, 내부에 9개 독립 Git 리포가 있어 중첩 리포 문제가 발생.

**문제점**:
- Git이 내부 리포를 서브모듈(gitlink)로 인식하려 함
- 원본 리포/워크트리 변경이 상위 `git status`에 노이즈로 표시
- `.git` 참조 구조가 꼬일 위험

**결정**: `.gitignore`로 개별 리포와 워크트리를 제외하고, 워크스페이스 고유 파일만 추적.

```gitignore
# 개별 리포지토리
hellobot-server/
hellobot_iOS/
...

# 워크트리
projects/*/worktrees/
```

**추적 대상**: CLAUDE.md, docs/, projects/(문서만), scripts/, .claude/commands/
**제외 대상**: 9개 원본 리포, 프로젝트 워크트리

**근거**: 워크스페이스 설정과 프로젝트 문서는 버전 관리가 필요하지만, 코드는 각 리포가 독립적으로 관리.

### 결정 13: 멀티 에이전트는 보류, 순차 실행으로 먼저 운영

**논의 배경**:
현재 에이전트 커맨드가 역할 프롬프트 방식(순차 실행)인데, Claude Code의 Agent 도구를 활용하면 서브에이전트를 병렬로 생성하여 진짜 멀티 에이전트 운영이 가능. 오케스트레이터 커맨드를 만들지 논의.

**결정**: 현재의 순차 실행 방식으로 먼저 시범 운영. 멀티 에이전트(오케스트레이터)는 보류.

**근거**: 실제로 돌려봐야 어디서 병목이 생기는지 알 수 있음. 프로젝트 문서 기반 커뮤니케이션, 워크트리 구조, 역할별 지침은 이미 갖춰져 있으므로 향후 멀티 에이전트로 전환 시 재활용 가능.

### 결정 9: 워크스페이스 관리자 에이전트 추가

**논의 배경**:
사용자가 "워크스페이스 레벨의 상태를 관리하는 역할이 필요하다"고 요청. 여러 피쳐가 동시에 진행되면 status.md가 실제와 달라지거나, 피쳐 목록에 누락이 생기거나, 핵심 문서(CLAUDE.md, architecture.md)가 현행과 맞지 않게 될 수 있음.

**결정**: `/workspace` 커맨드 추가. 워크스페이스 레벨 문서의 정합성 점검과 최신화를 담당.

**역할 범위**:
- 피쳐 목록(docs/features/readme.md)과 실제 디렉토리 대조
- 각 피쳐 status.md가 실제 진행과 일치하는지 검증
- CLAUDE.md, architecture.md 등 핵심 문서의 현행 일치 점검
- 오래된 문서 식별 및 최신화
- 코드는 건드리지 않음 — 문서 관리만 수행

**사용 패턴**:
- `/workspace` — 전체 점검
- `/workspace 피쳐명` — 특정 피쳐 문서만 점검
- 워크플로우의 특정 단계에 국한되지 않고, 필요할 때 수시로 호출

**근거**: 다른 에이전트들은 자기 파트에 집중하므로, 워크스페이스 전체의 문서 일관성을 유지할 전담 역할이 필요.
