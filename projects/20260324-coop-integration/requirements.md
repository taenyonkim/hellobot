# 요구사항 정의서: 쿠프마케팅 카카오 선물하기 상품권 연동

## 1. 연동 구조

```
카카오 선물하기 (구매자)
    ↓ 상품권 구매/발급
쿠프마케팅 (연동서버)
    ↓ 쿠폰인증 API (HTTP POST)
헬로우봇 (상품제공업체/Client)
    ↓ 상품 제공
최종 사용자 (쿠폰 수령자)
```

- **헬로우봇**: 상품제공업체 (문서상 "연동사", "Client")
- **쿠프마케팅**: 쿠폰 인증 중계 서버 운영
- **카카오**: 브랜드사 (상품권 판매)

## 2. 운영 상품

### 2-1. 하트 충전권

| 상품                | 금액     |
| ------------------- | -------- |
| 하트 충전권 5천원권 | 5,000원  |
| 하트 충전권 1만원권 | 10,000원 |

## 4. 헬로우봇 구현 요구사항

### 4-1. 핵심 기능

#### F1. 쿠폰 조회 (L0)

- 사용자가 쿠폰번호를 입력하면 쿠프마케팅 API로 쿠폰 상태 조회
- 조회 결과: 쿠폰명, 잔액, 사용여부, 쿠폰종류 등 표시
- 유효하지 않은 쿠폰(만료, 사용완료 등)은 사용자에게 안내 메시지
- 쿠프마케팅 응답의 `EndDay`는 **쿠폰 자체의 사용 기한**(카카오 선물하기 쿠폰 만료일)이며, 서버에서 사용 가능 여부 판단에만 사용. 교환된 상품의 유효기간과는 무관

#### F2. 쿠폰 사용 — 하트 충전권 (L1)

- 사용자가 하트 충전권 쿠폰을 입력 → 조회(L0) → 사용(L1) → 해당 상품에 정의된 하트 수량 충전
- CouponType `00`(교환권) — 금액별 별도 상품코드로 구분, AuthPrice 불필요
- 사용 성공 시 응답의 `BrandAuthCode` 저장 (취소 시 필요)
- **교환된 하트에는 유효기간 없음** — 충전 즉시 영구 사용 가능

#### F3. 쿠폰 사용 — 스킬 교환권 (L1)

- 사용자가 스킬 교환권 쿠폰을 입력 → 조회(L0) → 사용(L1) → 해당 스킬 100% 할인 쿠폰 발급
- 발급된 쿠폰으로 기존 스킬 구매 플로우를 통해 0하트 구매
- 교환권(00)은 1회 사용으로 소진
- 기존 쿠폰 시스템(CouponSpec/Coupon/CouponCondition) 활용
- **발급된 스킬 이용권에는 유효기간 없음** — 발급 즉시 영구 사용 가능

#### F4. 쿠폰 사용취소 (L2)

- **내부 오류 발생 시에만** 사용취소 처리 (L1 성공 후 하트 충전/쿠폰 발급 실패 시)
- 원거래 `BrandAuthCode` 필수
- **사용 후 취소 불가 정책**: 콘텐츠 제공형 상품이므로 정상 사용 후 사용자 요청에 의한 취소는 불가
- 고객 클레임 시: 쿠프마케팅에서 쿠폰번호 전달 → 정상 승인여부 확인하여 회신 (고객정보 마스킹)

#### F5. 망취소 처리 (L3)

- 쿠폰 사용/취소 요청 후 정상 응답을 받지 못한 경우 (타임아웃 등)
- 처리 순서: 조회(L0)로 쿠폰 상태 재확인 → 사용됨이면 망취소(L3) 호출 / 미사용이면 재시도 안내
- `OriginalAuthCode`(원거래 거래번호)로 롤백 처리

### 4-2. 데이터 관리

> 설계 상세: [backend-design.md](./backend-design.md) 섹션 2 (테이블 설계), 섹션 6 (Admin 페이지 설계)

#### D1. 쿠폰 사용 이력 테이블

- 쿠폰번호, 쿠폰종류, 처리유형, 사용금액, 승인번호, 거래번호
- 사용자 정보(userSeq), 처리 결과, 처리 일시
- 연동 상품 정보 (하트 충전권 / 상품 구매권 구분)
- API 호출 전문 로그 별도 관리 (트러블슈팅용)

#### D2. 운영 상품 관리

- 쿠프마케팅에서 등록한 상품 목록 (브랜드코드, 상품코드, 금액 등)
- 헬로우봇 내부 상품(하트/스킬)과의 매핑 관계

#### D3. 상품 관리 어드민

- 운영자가 쿠프마케팅 상품을 추가/수정/삭제할 수 있는 AdminJS 페이지
- 현재 판매중인 상품권 목록 조회 (활성/비활성 필터링)
- 상품 유형(하트충전권/스킬교환권)별 필드 관리
- super 권한 운영자만 생성/수정/삭제 가능

#### D3-1. 스킬 이용권(100% 할인 쿠폰) 자동 관리

- **생성**: 스킬 교환권 상품 생성 시 CouponSpec(PERCENTAGE 100%) + CouponCondition(skillSeqs) 자동 생성
- **수정**: 상품명/대상 스킬 변경 시 CouponSpec 이름/CouponCondition 조건 연동 업데이트
- **비활성화**: 상품 비활성화 시 CouponSpec.issueEndDate 설정 (신규 발급 중단, 기발급 유지)
- **재활성화**: 상품 활성화 시 CouponSpec.issueEndDate NULL로 복원 (발급 재개)
- 운영자가 CouponSpec을 직접 생성할 필요 없음 — 상품 CRUD와 연동되어 자동 관리

#### D4. 사용 내역 통계 (정산 검증)

- 기간별 상품권 사용/취소 통계 조회 (정산 대조용)
- 요약: 총 사용 건수, 총 취소 건수, 순 사용 건수, 금액 합계
- 상품별 내역: 상품코드별 사용/취소/순 건수 및 금액
- 일별 추이: 일자별 사용/취소/순 건수 및 금액
- 쿠프마케팅 정산서와 수치 대조하여 상호 검증

### 4-3. 처리 흐름 (하트 충전권 예시)

```
사용자: 쿠폰번호 입력
    ↓
[1] 쿠폰 조회 (L0)
    ↓ 유효성 확인 (유효기간, 사용여부, 쿠폰종류)
[2] 쿠폰 사용 (L1) + OriginalAuthCode 생성
    ↓ 성공 시 BrandAuthCode 저장
[3] 하트 충전 처리 (내부 로직)
    ↓
[4] 결과 표시
    ↓ (실패 시)
[5] 사용취소(L2) 또는 망취소(L3) 처리
```

### 4-4. 에러 핸들링

- 쿠프마케팅 API 호출 타임아웃 시 → 망취소(L3) 또는 조회(L0) 후 보상 처리
- 쿠폰 사용(L1) 성공 후 내부 하트 충전 실패 시 → 사용취소(L2)
- 모든 API 호출/응답 로깅 필수 (트러블슈팅 용도)

### 4-5. 보안

- 쿠폰번호 로그 마스킹
- 인증키(Auth_Key) 환경변수 관리 (`.env.json`)
- 쿠프마케팅 서버와의 통신은 상용 환경에서 HTTPS 확인 필요
