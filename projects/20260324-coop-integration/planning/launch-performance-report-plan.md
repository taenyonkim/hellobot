## 1. 의사결정 매트릭스 (스테이크홀더별)

성과 분석 리포트는 **하나가 아니다**. 의사결정 주체별로 보는 항목·주기·임계값이 다르다.

| 스테이크홀더 | 핵심 의사결정 | 보고 싶은 것 | 알 필요 없는 것 |
|---|---|---|---|
| **경영/사업 (CEO·사업책임)** | 채널 유지/확대/축소, 마케팅 투자 재배분 | 신규 구매자 추세, ROI(수수료 vs LTV), 채널 비교 | 시스템 헬스, 에러 분포 |
| **마케팅 운영 (MD·CRM)** | 상품 회수/재등록, 프로모션 타이밍·대상 | 상품별 판매 트렌드, 신규 vs 기존 구성, 등록 리드타임 | API 레이턴시 |
| **CS** | 사용자 문의 대응(쿠폰 등록 실패 등) | 에러 코드 분포, 구버전 가드 발동, 일별 실패 사례 | GMV·신규 구매자 수 |
| **개발/QA** | 핫픽스, 점진적 릴리스 결정 | 시스템 헬스(레이턴시·에러), 동시성 경합, BQ 데이터 무결성 | 매출 |
| **재무** | 매월 정산 처리 | 정산 대상 금액, 쿠프마케팅 L1 vs 내부 대사 결과 | 일간 트렌드 |

| # | 리포트 | 대상 | 주기 | 포맷 | 채널 | Producer | Reviewer |
|---|---|---|---|---|---|---|---|
| R1 | **출시 일일 모니터링** | 실무 전체 | 매일 09:30 KST (D+0~D+14) | Slack daily digest | `#kakao-gift-launch` (가칭) | 데이터팀 자동 발송 | PM (이상 감지 시 점검) |
| R2 | **출시 1주 회고** | PM·사업·데이터·개발 | D+7 1회 | 슬라이드 30분 | 정기 회의 | PM | 사업 |
| R3 | **주간 성과 리포트** | 사업·마케팅·PM | 매주 월 10:00 KST (D+15부터) | Slack 요약 + Looker 링크 | `#kakao-gift-performance` (가칭) | 데이터팀 자동 발송 | 사업 |

## 3. 리포트별 구성

### R1. 출시 일일 모니터링 (D+0 ~ D+14)

**목적**: 출시 직후 시스템 안정성 + 시장 진입 즉시 신호 포착.

**구성** (Slack 단일 메시지):
```
🟢 카카오 선물하기 D+N 일일 리포트 (YYYY-MM-DD)

[등록]   누적 N건 (어제 +N건, 7일 평균 +N건)
[성공률] 99.X% (실패 N건 / 시도 N건)
[가드]   구버전 가드 N건, Redlock 경합 N건
[매출]   누적 ₩N (어제 +₩N) ※ 카카오 GMV 추정치
[신규]   카카오 신규 구매자 N명 (D+1 기준)
[Top3]   상품별 등록: ProductA(N), B(N), C(N)
[알림]   ⚠️ CM_005 30건/일 임계 초과 — 점검 필요

📊 상세: <Looker 링크>
```

**의사결정 게이트**:
| 조건 | 액션 |
|------|------|
| 등록 성공률 < 95% | 즉시 점검 회의 (개발) |
| 가드 발동 > 100건/일 | 강제 업데이트 푸시 검토 (PM) |
| 등록 0건 7일 연속 | 카카오 노출 점검 + 쿠프마케팅 측 협의 |
| L1 API 5xx > 5%/일 | 쿠프마케팅 측 장애 신고 |

**구현 의존성**:
- 운영 지표(가드·레이턴시)는 현재 운영 대시보드 미구축 → **출시 전 BQ 임시 쿼리 + 수동 일일 발송**으로 시작 가능 (Looker 정식판은 D+30 내 구축).
- 신규 구매자 수치는 Q4 컬럼(`coop_kakao_first_used_date`) 4/30 구현 후 가능.

---

### R2. 출시 1주 회고 (D+7)

**목적**: 출시 첫 주 데이터로 가설 검증 + Phase 2 준비 결정.

**Section**:
1. 가설 점검 (1pager `Success Metric` 대비)
   - 신규 구매자 +20명/일 KPI 달성 여부
   - 등록 전환율(쿠프 발급 → 헬로우봇 등록) 첫 측정값
2. 시스템 헬스 종합 (R1의 7일 누적)
3. 상품 성과 (라인업별 등록 분포, 스킬 교환권 소진율)
4. 사용자 행동 (등록 후 24h 내 추가 구매 여부, R1 등록자의 ARPPU)
5. 문제·이슈 정리 (CS 인입 사례, 가드 발동 패턴)
6. **결정 사항**: 노출 확대/축소, 상품 라인업 조정, 후속 프로모션

**의사결정 임계값** (사용자 확정 필요):
- ✅ Go: 일평균 등록 ≥ 20건, 성공률 ≥ 99%, 신규 구매자 비율 ≥ 50%
- ⚠️ Watch: 위 조건 일부 미달 → 2주차 추가 관찰
- ❌ Pivot: 시스템 안정 but 등록 5건/일 미만 → 카카오 노출 재협의 필요

---

### R3. 주간 성과 리포트 (D+15부터)

**목적**: 안정 운영기 진입 후 정기 성과 추적. 의사결정보다 **추세 모니터링**이 주.

**Section** (Slack 요약 + Looker 상세):
1. 주간 KPI 카드 (지난주 대비 ▲/▼)
   - 신규 구매자 (앱/웹 분리)
   - 등록 건수
   - 등록 전환율 (입수 가능 시)
   - GMV
2. 주간 Top 상품 변동
3. 이상치 알림 (전주 대비 ±30% 변동 자동 플래그)
4. Looker 링크 (4개 대시보드 — 경영/운영/상품/정산)

---

### R4. 월간 성과 리뷰 (M+5영업일)

**목적**: 경영진 의사결정용. 채널 ROI·라인업·확장.

**Section** (슬라이드 ~15p):
1. 한 줄 요약 (이번 달 채널 성과 + 의사결정 제안)
2. North Star: 카카오 채널 신규 구매자 MAU
3. ROI 분석
   - 매출 (Q1 인젝션 기반)
   - 수수료 (8% 차감)
   - 신규 구매자 LTV 추정 (vs 다른 채널)
   - 채널 단위경제 (CAC = 수수료 / 신규 구매자, vs Meta Paid CAC)
4. 입력 지표 (전환율, 리드타임, 등록 성공률)
5. 상품 성과 (라인업 Top/Bottom, 신규 라인업 후보)
6. 운영 리스크 (CS 인입 추이, 시스템 안정성)
7. 다음 달 의제 (라인업 변경, 프로모션 캘린더, 채널 확장 제안)

**Owner**: PM 작성 + 사업 검토 + CEO 컨펌.

---

### R5. 분기 회고 (Q+15일)

**목적**: 1pager Customer Job("지속 가능한 신규 채널 발굴") 충족 여부 평가.

**Section** (슬라이드 ~25p):
1. 분기 누적 KPI vs 1pager Success Metric
2. 채널 비교: 카카오 vs Meta Paid vs 오가닉
   - CAC, ROAS, LTV 곡선
   - 신규 구매자 점유율
3. 학습 포인트 (잘된 것 / 못된 것 / 의외)
4. 지속 가능성 평가 (1pager Problem 해결 여부)
5. **다음 분기 결정**: 채널 확대(타 플랫폼 제휴), 상품 라인업 전략, 운영 리소스

## 5. 데이터 인프라 ↔ 리포트 의존성

리포트 발행 가능 시점은 **데이터 인프라 준비 일정에 종속**. 현재 갭:

| 필요 데이터 | 현재 상태 | 리포트 영향 |
|---|---|---|
| 카카오 신규 구매자 분류 | Q4 4/30 구현 (architecture.md §10-7) | R2 D+7부터 가능. R1 D+1 신규 수치는 4/30 이후 |
| 매출(GMV) — Q1 인젝션 | 서버 구현 완료, 배포·BQ 검증 대기 | 출시 시점부터 가능 (단, 검증 통과 전제) |
| 등록 건수·성공률 | 서버 이벤트 3종 미구현 (tasks.md §155) | **출시 전 결정 필요 — Phase 1 출시에 포함할지** |
| 에러 코드 분포 | 동상 | 동상 |
| 구버전 가드 발동 | 서버 메트릭 미수집 (tasks.md #11) | 출시 전 수집 시작 권장 |
| L1 API 레이턴시 | `coop_marketing_api_log` 적재 중 (BQ 마트 미구축) | R2부터 가능 |
| 등록 전환율 분모 (쿠프 발급) | Q3 결정에서 인입 미구현 (불필요 판정) | 발급 데이터 없으면 측정 불가 → **재논의 필요** |
| 쿠프마케팅 정산 대사 | DAG 미구현 | R4부터 필요 — 첫 정산(M+10) 전 구축 |
| Looker 대시보드 | 도구 미정 | R3·R4 본격 운영 전 결정 |

### 5.1 재논의 필요 — Q3 결정 보강

Q3에서 "정산 데이터 인입 불필요"로 결정했으나, **등록 전환율 (쿠프 발급 → 헬로우봇 등록)** 측정에는 발급 데이터가 필수. 재정의 옵션:

| 옵션 | 트레이드오프 |
|------|------------|
| A. 등록 전환율 KPI 폐기 | 1pager Input Metric "상품권 구매 전환율" 직접 측정 불가 |
| B. 월 1회 쿠프 발급 CSV 수동 인입 | 운영 부담, but 충분 (월간 리뷰용) |
| C. 일일 CSV/API 자동 인입 (Q3 부분 부활) | DAG 구축 필요, 정확한 일일 모니터링 가능 |

**권장**: B — 월간 리뷰 R4용으로만 발급 데이터를 수령. 일일 R1·주간 R3에서는 "내부 등록 건수"만 다루고 전환율은 제외.
