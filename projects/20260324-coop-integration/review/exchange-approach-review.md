# 상품권 교환 방식 비교 검토

## 검토 목적

카카오 선물하기 상품권(하트충전권, 개별스킬 교환권) 등록 후 교환 방식을 두 가지 관점에서 비교 분석

## 방식 정의

### 방식 1: 즉시 교환

```
사용자가 상품권 코드 입력
  → 쿠프마케팅 API L1 (사용 처리)
  → 즉시 상품 제공 (하트 충전 또는 스킬 해금)
  → 이력 기록
```

- 기프티엘 연동과 동일한 패턴
- 상품권 등록 = 상품 교환이 한 번에 이루어짐

### 방식 2: 쿠폰 발행 후 교환

```
사용자가 상품권 코드 입력
  → 쿠프마케팅 API L1 (사용 처리)
  → 상품권과 동일 조건의 내부 쿠폰 발행
  → (사용자가 원할 때) 쿠폰 사용으로 상품 교환
```

- 상품권 등록과 상품 교환이 분리됨
- 내부 쿠폰 시스템을 중간 매개로 활용

---

## 기존 시스템 현황 요약

### 내부 쿠폰 시스템 (CouponSpec/Coupon/CouponCode)

| 항목 | 현재 상태 |
|------|----------|
| 설계 목적 | **결제 할인** (PERCENTAGE/FIXED 할인) |
| 쿠폰 타입 | AUTO, DOWNLOAD, SINGLE_CODE, MULTI_CODE |
| 할인 방식 | 스킬 구매 시 가격 할인 적용 |
| 적용 시점 | Payment(결제) 플로우에서만 사용 |
| 조건 필터링 | CouponCondition으로 특정 스킬 타겟팅 가능 |
| 하트 충전 지원 | **없음** — 스킬 구매 할인 전용 |

### 스킬 접근 방식 (PremiumSkillLog)

| 항목 | 현재 상태 |
|------|----------|
| 스킬 해금 판단 | `PremiumSkillLogType.Payment` 존재 여부로 판단 |
| 해금 타입 | `Payment`, `Reset` 두 가지만 존재 |
| 무료 해금 | **직접 지원 불가** — Payment 로그 생성 필요 |

### HeartService

| 항목 | 현재 상태 |
|------|----------|
| chargeHeart() | redlock + SERIALIZABLE 트랜잭션으로 안전한 충전 |
| HeartLogType | ChargeByPurchase, ChargeByAdmin, ChargeByPromotionEvent 등 10종 |
| 외부 연동 충전 | 전용 타입 없음 (새 타입 추가 필요) |

## 상품 유형별 비교 분석

### A. 하트 충전권

#### 방식 1: 즉시 교환

```
상품권 등록 → L1 사용 → chargeHeart() → 완료
```

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | ✅ **낮음** — CoupcMarketingService + HeartService.chargeHeart() 호출 |
| 신규 코드 | CoupcMarketingService, CoupcMarketingCouponUsage 엔티티 |
| 기존 시스템 수정 | HeartLogType 1개 추가만 필요 |
| 사용자 경험 | 코드 입력 → 바로 하트 충전 (1단계) |
| 실패 복구 | L1 성공 후 chargeHeart() 실패 시 → L3 망취소로 복구 |
| 동시성 | chargeHeart()의 redlock이 보장 |

#### 방식 2: 쿠폰 발행 후 교환

```
상품권 등록 → L1 사용 → 내부 쿠폰 발행 → (이후) 쿠폰으로 하트 충전
```

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | ❌ **높음** — 쿠폰 시스템 대폭 확장 필요 |
| 신규 코드 | CoupcMarketingService + 쿠폰 시스템 확장 + 하트 충전 쿠폰 사용 로직 |
| 기존 시스템 수정 | ⚠️ **대규모** — 아래 상세 |
| 사용자 경험 | 코드 입력 → 쿠폰 확인 → 쿠폰 사용 → 하트 충전 (3단계) |
| 실패 복구 | 쿠폰 발행까지만 성공하면 이후 재시도 가능 |
| 동시성 | 쿠폰 사용 시 chargeHeart()의 redlock 적용 필요 |

**쿠폰 시스템 확장 필요 사항**:

1. 현재 쿠폰은 **결제 할인** 전용 → "하트 충전" 기능이 없음
2. CouponSpec에 새로운 용도 (discountType 외에 "HEART_CHARGE" 등) 추가 필요
3. 쿠폰 사용 플로우에 하트 충전 분기 추가 필요 (couponService.useCoupon() 수정)
4. 하트 수량 매핑 로직 필요 (금액 → 하트 환산)
5. 쿠폰 목록 UI에서 "하트 충전 쿠폰"을 별도로 표시해야 함
6. **기존 결제 할인 쿠폰과 하트 충전 쿠폰의 혼재** → 클라이언트/서버 양쪽에서 분기 처리 증가

#### 하트 충전권 결론: ✅ 방식 1 권장

기존 쿠폰 시스템이 "결제 할인" 목적이라 "하트 직접 충전"을 끼워넣으려면 근본적인 확장이 필요. chargeHeart()가 이미 안전한 충전 메커니즘을 제공하므로 즉시 교환이 자연스럽고 구현 비용이 크게 낮음.

### B. 개별 스킬 교환권

#### 방식 1: 즉시 교환

```
상품권 등록 → L1 사용 → 스킬 해금 (PremiumSkillLog 생성) → 완료
```

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | ✅ **낮음** — CoupcMarketingService + PremiumSkillLog 생성 |
| 신규 코드 | CoupcMarketingService, CoupcMarketingProduct(상품-스킬 매핑), CoupcMarketingCouponUsage |
| 기존 시스템 수정 | PremiumSkillLogType에 새 타입 추가 + isPremiumSkillPaid() 수정 필요 |
| 사용자 경험 | 코드 입력 → 바로 스킬 이용 가능 (1단계) |
| 실패 복구 | L1 성공 후 스킬 해금 실패 시 → L3 망취소로 복구 |
| 기프티엘 패턴 | 거의 동일한 패턴으로 검증됨 |

**PremiumSkillLog 확장 필요 사항**:
- `PremiumSkillLogType`에 `GiftCoupon = "GIFT_COUPON"` 같은 새 타입 추가
- `isPremiumSkillPaid()` 로직에서 `GiftCoupon` 타입도 "사용됨"으로 인정하도록 수정

#### 방식 2: 쿠폰 발행 후 교환

```
상품권 등록 → L1 사용 → 해당 스킬 100% 할인 쿠폰 발행 → (이후) 스킬 구매 시 쿠폰 적용
```

| 항목 | 평가 |
|------|------|
| 구현 복잡도 | ⚠️ **중간** — 기존 쿠폰 시스템과 비교적 정합 |
| 신규 코드 | CoupcMarketingService + CouponSpec/Coupon 생성 로직 |
| 기존 시스템 수정 | CouponCondition에 스킬 타겟팅 이미 존재하므로 비교적 적음 |
| 사용자 경험 | 코드 입력 → 쿠폰 확인 → 스킬 페이지에서 쿠폰 적용하여 무료 구매 (3단계) |
| 실패 복구 | 쿠폰 발행까지만 성공하면 이후 재시도 가능 |
| 기존 UI 활용 | 쿠폰 목록, 스킬 구매 화면에서 쿠폰 표시 가능 |

**쿠폰 기반 스킬 교환의 자연스러운 점**:
- 기존 CouponSpec의 `FIXED` 할인으로 100% 할인 (= 무료 구매) 표현 가능
- CouponCondition의 skillSeqs 필터로 특정 스킬에만 적용 가능
- 기존 결제 플로우(Payment)에서 쿠폰 적용 로직 재사용 가능

**쿠폰 기반 스킬 교환의 문제점**:
- 사용자가 스킬 교환권을 등록했는데 **바로 스킬이 제공되지 않음** → UX 혼란
- 스킬 구매 페이지로 이동 → 쿠폰 선택 → 구매 확인이라는 추가 단계 필요
- "선물하기"로 받은 상품권인데 "구매" 화면을 거쳐야 하는 인지 부조화
- 100% 할인 쿠폰인데 maxDiscountPrice 검증 등 기존 로직과의 엣지케이스 발생 가능
- 쿠폰 유효기간 관리가 이중 (쿠프마케팅 유효기간 + 내부 쿠폰 유효기간)

#### 스킬 교환권 결론: ✅ 방식 1 권장 (방식 2도 차선으로 가능)

스킬 교환권은 기존 쿠폰 시스템과 형태가 유사하여 방식 2도 기술적으로 가능하나, **UX 관점에서 "상품권 등록 → 즉시 사용 가능"이 자연스러움**. 기프티엘에서 이미 검증된 즉시 교환 패턴 활용이 더 효율적.

## 종합 비교

| 비교 항목 | 방식 1 (즉시 교환) | 방식 2 (쿠폰 발행 후 교환) |
|----------|-------------------|--------------------------|
| **구현 복잡도** | ✅ 낮음 | ❌ 높음 (특히 하트충전권) |
| **기존 시스템 수정** | 최소 (타입 추가 수준) | 대규모 (쿠폰 시스템 확장) |
| **UX (사용자 단계)** | ✅ 1단계 | ❌ 3단계 |
| **UX (직관성)** | ✅ "등록 → 바로 사용" | ⚠️ "등록 → 쿠폰 → 사용" 혼란 |
| **실패 복구** | L3 망취소로 원복 | ⚠️ 쿠폰 발행 후 재시도 가능하나 관리 복잡 |
| **검증된 패턴** | ✅ 기프티엘 동일 패턴 | 신규 패턴 |
| **클라이언트 개발** | ✅ 상품권 등록 화면만 | 쿠폰 목록/구매 화면 수정 |
| **데이터 관리** | 단일 이력 (CoupcMarketingCouponUsage) | 이중 이력 (상품권 로그 + 쿠폰 로그) |
| **사용 유연성** | ⚠️ 등록 즉시 교환 (보관 불가) | ✅ 원하는 시점에 사용 가능 |
| **쿠폰함 노출** | ❌ 별도 UI 필요 | ✅ 기존 쿠폰함에 통합 가능 |

## 방식 2가 유리한 시나리오

방식 2가 의미 있으려면 다음 조건을 만족해야 함:

1. **사용 시점 분리가 필수 요구사항인 경우**: 상품권을 등록해놓고 나중에 원하는 때에 사용하고 싶은 경우
2. **쿠폰함 통합이 핵심 기획인 경우**: 기존 쿠폰함에서 상품권도 함께 관리하는 것이 기획 의도인 경우
3. **다양한 스킬 중 선택이 필요한 경우**: "아무 스킬이나 1개 교환" 같은 범용 교환권 → 쿠폰으로 발행 후 사용자가 스킬 선택

현재 요구사항(카카오 선물하기 = 특정 상품에 매핑된 상품권)에서는 위 조건에 해당하지 않을 가능성이 높음.

## 최종 권장

### ✅ 방식 1 (즉시 교환) 권장

| 근거 | 설명 |
|------|------|
| 구현 효율 | 기프티엘 패턴을 그대로 참조하여 신규 서비스 작성 (검증된 패턴) |
| 시스템 영향 최소 | 기존 쿠폰 시스템을 건드리지 않아 사이드이펙트 없음 |
| UX 자연스러움 | "선물 받음 → 코드 입력 → 바로 사용" 직관적 흐름 |
| 안정성 | L3 망취소로 실패 케이스 원복 가능 |
| 클라이언트 비용 | 상품권 등록 화면만 신규 개발, 기존 화면 수정 불필요 |

### 구현 시 상품 유형별 분기

```
상품권 코드 입력
  → 쿠프마케팅 L0 (조회)
  → 상품 유형 확인

  [하트 충전권]
    → L1 (사용) → HeartService.chargeHeart() → 완료

  [스킬 교환권]
    → L1 (사용) → PremiumSkillLog 생성 → 스킬 해금 → 완료
```

### 향후 방식 2 전환이 필요해질 경우

만약 "상품권을 보관했다가 나중에 사용" 기능이 기획 요구사항으로 추가되면, 기존 쿠폰 시스템을 확장하는 대신 **별도의 상품권 보관함(Gift Vault)** 개념으로 분리 구현하는 것이 기존 시스템 영향을 최소화하는 방향.

## 최종 확정 (2026-04-06)

### 하이브리드 방식 채택

검토 결과와 실제 코드 분석을 거쳐, 상품 유형별로 다른 방식을 적용하기로 확정했습니다:

| 상품 유형 | 방식 | 설명 |
|----------|------|------|
| **하트 충전권** | 방식 1 (즉시 교환) | L1 → HeartService.chargeHeart() → 즉시 충전 |
| **스킬 교환권** | 방식 2 변형 (100% 할인 쿠폰 발급) | L1 → 100% 할인 쿠폰 발급 → 기존 스킬 구매 플로우로 0하트 구매 |

### 스킬 교환권에 쿠폰 발급 방식을 채택한 이유

1. **기존 구매 인프라 재사용**: Payment, PremiumSkillLog, 쿠폰 할인 계산 등 기존 스킬 구매 플로우를 그대로 활용
2. **이후 플로우 처리 용이**: 구매/취소/환불 등 후속 처리가 기존 시스템과 동일하게 동작
3. **쿠폰 시스템과의 높은 정합성**: 기존 CouponSpec(PERCENTAGE 100%) + CouponCondition(skillSeqs)으로 자연스럽게 표현 가능
4. **유연성**: 쿠폰 발급 후 사용자가 원하는 시점에 구매 가능, 쿠폰함에서 관리 가능
5. **취소 단순화**: 쿠폰 미사용 시 쿠폰 회수로 단순 처리, 사�� 완료 시 기존 구매 취소 로직 활용

### 하트 충전권은 즉시 교환을 유지��는 이유

- 기존 쿠폰 시스템이 "결제 할인" 전용이라 "하트 직접 충전"은 표현 불가
- chargeHeart()가 이미 안전한 충전 메커니즘 제공 (redlock + SERIALIZABLE)
- 즉시 충전이 UX적으로 자연스러움

### 초기 검토와의 차이점

- 초기 검토에서는 스킬 교환권도 방식 1(즉시 교환)을 권장했으나, 코드 분석 결과 기존 쿠폰 시스템이 스킬 100% 할인을 이미 완벽히 지원하�� 있어 방식 2 변형이 더 효율적
- 방식 2의 주요 단점이었던 "UX 혼란(3단계)"은 S4 팝업에서 스킬 구매 페이지로 자연스럽게 연결하여 해소
