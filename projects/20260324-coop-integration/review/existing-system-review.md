# 기존 쿠폰 시스템 활용 검토

## 검토 목적
쿠프마케팅 카카오 선물하기 상품권(하트충전권, 개별스킬 교환권) 구현 시 기존 쿠폰 시스템을 재사용할 수 있는지 검토

## 기존 쿠폰 관련 시스템 현황

### 1. 내부 할인 쿠폰 시스템 (Coupon/CouponSpec/CouponCode)
| 항목 | 내용 |
|------|------|
| 용도 | 결제 시 할인 적용 (PERCENTAGE/FIXED) |
| 핵심 엔티티 | `CouponSpec`(쿠폰 템플릿), `Coupon`(발급 인스턴스), `CouponCode`(코드), `CouponCondition`(타겟팅) |
| 처리 방식 | 내부 발급/사용, 외부 API 연동 없음 |
| 사용 흐름 | 쿠폰 발급 → 결제 시 할인 적용 → Payment에 기록 |

### 2. 기프티엘(Giftiel) 연동 시스템 ⭐
| 항목 | 내용 |
|------|------|
| 용도 | 외부 상품권 → 스킬(콘텐츠) 교환 |
| 핵심 엔티티 | `GiftielProduct`(상품-스킬 매핑), `GiftielCouponLog`(사용 이력) |
| 처리 방식 | 외부 API 호출 (L0 조회, L1 사용, L2 사용취소) |
| 사용 흐름 | 쿠폰 조회(L0) → 사용(L1) → 스킬 제공 → 이력 기록 |

### 3. Cafe24 쿠폰 시스템
| 항목 | 내용 |
|------|------|
| 용도 | 하트 구매 시 실물 쇼핑몰 쿠폰 지급 (사은품) |
| 처리 방식 | 내부 쿠폰 재고 관리 + 카카오톡 알림톡 발송 |

### 4. 하트(Heart) 시스템
| 항목 | 내용 |
|------|------|
| 핵심 엔티티 | `HeartLog`, `HeartLogDetail` |
| 충전 타입 | `ChargeByPurchase`, `ChargeByAdmin`, `ChargeByPromotionEvent` 등 |
| 충전 메서드 | `HeartService.chargeHeart()` — redlock + SERIALIZABLE 트랜잭션 |

## 쿠프마케팅 vs 기프티엘 비교 분석

쿠프마케팅 API는 기프티엘과 **거의 동일한 프로토콜 구조**를 사용합니다.

| 비교 항목 | 기프티엘 (기존) | 쿠프마케팅 (신규) |
|-----------|----------------|------------------|
| API 방식 | HTTP POST (단일 엔드포인트) | HTTP POST (단일 엔드포인트 `/AuthUse`) |
| 조회 명령 | `L0` | `L0` |
| 사용 명령 | `L1` | `L1` |
| 취소 명령 | `L2` | `L2` |
| 망취소 | 없음 | `L3` (추가) |
| 인증 방식 | `COMP_CODE` + `BRANCH_CODE` | `Auth_Key` + `CompCode` + `BranchCode` |
| 쿠폰 타입 | `00` (교환권 고정) | `00`/`01`/`02`/`03` (다중 지원) |
| 금액 처리 | 없음 (`USE_PRICE: ""`) | `AuthPrice` (잔액관리형 시 필수) |
| 상품 매핑 | `PRODUCT_CODE` → `GiftielProduct` | 상품코드 → 스킬/하트 매핑 필요 |
| 성공 코드 | `RESULTCODE: "00"` | `ResultCode: "0000"` |
| 응답 필드 | 15개 | 20개+ (더 상세) |

## 시스템별 재사용 가능성 평가

### A. 개별스킬 교환권 → 기프티엘 시스템 재사용 검토

#### 기프티엘 구조 요약
```
GiftielService.checkCoupon()  → L0 → 유효성 확인 + 상품 매핑
GiftielService.useCoupon()    → L1 → 사용 처리 + 이력 기록
GiftielService.unuseCoupon()  → L2 → 사용취소 + 이력 기록
GiftielProduct                → fixedMenuSeq로 스킬 매핑
GiftielCouponLog              → (userSeq, couponCode) 유니크 이력
```

#### 재사용 가능한 부분
- **처리 흐름**: L0 → L1 → 스킬 제공 패턴이 거의 동일
- **상품-스킬 매핑**: `GiftielProduct` 같은 매핑 테이블 구조 참조 가능
- **이력 관리**: `GiftielCouponLog`의 (userSeq, couponCode, isCanceled) 패턴 참조 가능

#### 직접 재사용이 어려운 부분
| 항목 | 이유 |
|------|------|
| `GiftielService` 코드 | API 엔드포인트, 인증 방식, 요청/응답 필드가 다름 |
| `GiftielProduct` 테이블 | 기프티엘 전용 (snapshotFixedMenu/snapshotBlock 관계) |
| `GiftielCouponLog` 테이블 | 기프티엘 전용, 쿠프마케팅은 추가 필드 필요 (승인번호, 금액 등) |
| 에러 처리 | 기프티엘(`"00"` 성공)과 쿠프마케팅(`"0000"` 성공)이 다름 |
| 망취소(L3) | 기프티엘에는 없는 기능, 신규 구현 필요 |

#### 결론: ❌ 직접 재사용 불가, ✅ 패턴 참조하여 신규 구현

---

### B. 하트 충전권 → 하트 시스템 재사용 검토

#### HeartService 재사용 가능 부분
```typescript
// 기존 chargeHeart() 메서드 — 그대로 호출 가능
await heartService.chargeHeart({
  requestContext,
  logType: HeartLogType.ChargeByPromotionEvent, // 또는 새 타입 추가
  description: "카카오 선물하기 하트 충전권 5,000원",
  quantity: 500, // 환산된 하트 수량
  expiredAt: null,
  data: { ... },
});
```

#### 검토 사항
| 항목 | 평가 | 설명 |
|------|------|------|
| `chargeHeart()` 메서드 | ✅ 재사용 | redlock + 트랜잭션 처리 그대로 활용 |
| `HeartLogType` | ⚠️ 새 타입 추가 권장 | `ChargeByGiftCoupon` 등 추적 목적으로 구분 필요 |
| `HeartLogData` | ⚠️ 확장 필요 | 쿠폰번호, 승인번호 등 메타데이터 저장 필드 추가 |
| 금액 → 하트 환산 | ✅ 확정 | 상품 정의 테이블(coupc_marketing_product)에서 상품코드별 관리 |
| 잔액관리형 부분사용 | ✅ 확정: 미지원 | 전액 1회 소진만 지원 (부분 사용 불필요) |

#### 결론: ✅ `HeartService.chargeHeart()` 재사용, 나머지는 신규 구현

### C. 내부 할인 쿠폰 시스템 (Coupon/CouponSpec) 활용 가능성

| 항목 | 평가 | 설명 |
|------|------|------|
| 설계 목적 | ❌ 불일치 | 결제 할인용이며, 외부 상품권 교환용이 아님 |
| CouponSpec | ❌ 불필요 | 발급 수량/기간/할인율 등 불필요한 필드가 대부분 |
| CouponCode | ❌ 불일치 | 내부 코드 관리용, 외부 쿠폰번호 인증 용도가 아님 |
| CouponCondition | ❌ 불필요 | 타겟팅 조건은 외부 상품권과 무관 |

#### 결론: ❌ 활용 불가 — 설계 목적 자체가 다름

## 최종 검토 결과

### 권장 구현 방향

```
┌─────────────────────────────────────────────────────┐
│              CoupcMarketingService (신규)             │
│                                                     │
│  - requestApi()        → 쿠프마케팅 API 호출 (공통)    │
│  - checkCoupon()       → L0 조회                     │
│  - useCoupon()         → L1 사용                     │
│  - cancelCoupon()      → L2 사용취소                  │
│  - networkCancel()     → L3 망취소                    │
├─────────────────────────────────────────────────────┤
│                   분기 처리                           │
│                                                     │
│  교환권(00) → 스킬 제공 로직 (기프티엘 패턴 참조)       │
│  잔액관리형(02/03) → HeartService.chargeHeart() 호출  │ ← 재사용
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│           신규 엔티티                                 │
│                                                     │
│  CoupcMarketingProduct      → 상품-스킬/하트 매핑     │
│  CoupcMarketingCouponUsage  → 상품권 사용 이력        │
│  CoupcMarketingApiLog       → API 호출 전문 로그      │
└─────────────────────────────────────────────────────┘
```

### 요약 매트릭스

| 구분 | 기존 시스템 | 재사용 여부 | 비고 |
|------|-----------|:----------:|------|
| **스킬 교환권** | 기프티엘(Giftiel) | 패턴만 참조 | API 구조 유사, 코드 직접 재사용 불가 |
| **하트 충전권** | HeartService | ✅ chargeHeart() 재사용 | HeartLogType 추가, HeartLogData 확장 필요 |
| **내부 쿠폰** | Coupon/CouponSpec | ❌ 미사용 | 할인 쿠폰이라 용도가 다름 |
| **API 호출** | Giftiel 패턴 참조 | 패턴만 참조 | axios + config 구조 참조 |
| **동시성 제어** | redlock 패턴 | ✅ 재사용 | 기존 패턴 그대로 적용 |
| **에러 처리** | HttpError 패턴 | ✅ 재사용 | ErrorCode 추가 필요 |

### 신규 구현 필요 목록

1. **CoupcMarketingService** — 쿠프마케팅 API 연동 서비스 (기프티엘 서비스 패턴 참조)
2. **CoupcMarketingProduct** — 상품-스킬/하트 매핑 엔티티
3. **CoupcMarketingCouponUsage** — 상품권 사용 이력 엔티티 (승인번호, 거래번호, 상태 등)
4. **CoupcMarketingApiLog** — API 호출 전문 로그 엔티티 (요청/응답 JSONB)
5. **CoupcMarketingController** — API 엔드포인트
6. **DTO** — 요청/응답 DTO
7. **config 추가** — 쿠프마케팅 URL, 인증키, 코드 등
8. **HeartLogType 추가** — `ChargeByGiftCoupon` (하트 충전권 구분용)
9. **ErrorCode 추가** — 쿠프마케팅 에러 코드 매핑
10. **망취소(L3) 로직** — 기프티엘에 없는 신규 기능
11. **Admin 페이지** — 상품 관리 CRUD, 사용 이력 조회(+취소 액션), API 로그 조회
