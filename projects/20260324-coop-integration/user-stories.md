# 사용자 스토리

## 목적

쿠프마케팅 카카오 선물하기 상품권 연동의 주요 사용자 스토리를 정의하고, 각 스토리별로 구체화가 필요한 부분과 쿠프마케팅에 확인해야 할 사항을 식별합니다.

---

## 1. 사용자 스토리

### US-1. 하트 충전권 사용

> 카카오 선물하기로 하트 충전권을 받은 사용자가, 헬로우봇에서 쿠폰번호를 입력하여 즉시 하트를 충전받는다.

**사전 조건**:
- 사용자가 헬로우봇에 로그인한 상태
- 카카오 선물하기에서 수령한 유효한 하트 충전권 쿠폰번호를 보유

**기본 흐름**:
1. 사용자가 프로필 → 쿠폰 탭에 진입한다 (또는 카카오 딥링크로 직접 진입)
2. 기존 쿠폰 입력란에 상품권 코드를 입력하고 "등록"을 탭한다
3. 서버가 코드를 판별하여 쿠프마케팅 조회(L0) → 하트 충전권 확인 팝업을 표시한다 (상품명, 충전될 하트 수량)
4. 사용자가 "사용하기"를 누른다
5. 시스템이 쿠폰 사용(L1) 처리 후 하트를 충전한다
6. 충전 완료 팝업 → "충전 확인하기" → **내 하트 페이지**로 이동하여 충전 결과 확인

**예외 흐름**:
- 3a. 만료된 쿠폰 → "기간이 만료된 쿠폰이에요" 안내
- 3b. 이미 사용된 쿠폰 → "이미 사용된 쿠폰이에요" 안내
- 3c. 존재하지 않는 쿠폰 → "유효하지 않은 쿠폰번호예요" 안내
- 5a. 쿠프마케팅 API 타임아웃 → 망취소(L3) 처리, "일시적 오류가 발생했어요. 다시 시도해주세요" 안내
- 5b. L1 성공 후 하트 충전 실패 → 사용취소(L2) 롤백, "오류가 발생하여 쿠폰이 원복되었어요" 안내

---

### US-2. 스킬 교환권 사용

> 카카오 선물하기로 스킬 교환권을 받은 사용자가, 헬로우봇에서 쿠폰번호를 입력하면 해당 스킬의 100% 할인 쿠폰이 발급되고, 기존 스킬 구매 플로우를 통해 0하트로 스킬을 구매할 수 있다.

**사전 조건**:
- 사용자가 헬로우봇에 로그인한 상태
- 카카오 선물하기에서 수령한 유효한 스킬 교환권 쿠폰번호를 보유

**기본 흐름**:
1. 사용자가 프로필 → 쿠폰 탭에 진입한다 (또는 카카오 딥링크로 직접 진입)
2. 기존 쿠폰 입력란에 상품권 코드를 입력하고 "등록"을 탭한다
3. 서버가 코드를 판별하여 쿠프마케팅 조회(L0) → 스킬 교환권 확인 팝업을 표시한다 (스킬명, 배너 이미지, 챗봇명)
4. 사용자가 "사용하기"를 누른다
5. 시스템이 쿠폰 사용(L1) 처리 후 해당 스킬의 100% 할인 쿠폰("스킬 이용권")을 발급한다
6. 팝업 닫힘 → 토스트 "스킬 이용권이 등록되었어요" → **쿠폰 리스트 최상단에 스킬 이용권 카드 추가**
7. 이용권 카드 탭 → **스킬 상세 페이지** (쿠폰 적용, ♥ 0 표시) → "지금 보러가기" → **챗봇 채팅방** (스킬 자동 시작)

**예외 흐름**:
- US-1과 동일한 쿠폰 관련 예외
- 5c. L1 성공 후 쿠폰 발급 실패 → 사용취소(L2) 롤백
- 7a. 이미 보유한 스킬인 경우 → 기존 스킬 구매 중복 체크 로직에 의해 처리 (쿠폰은 보관)

---

### US-3. 쿠폰 상태만 확인

> 사용자가 쿠폰번호를 입력하여 해당 쿠폰이 유효한지, 어떤 상품인지 확인한다 (사용 전 미리보기).

**흐름**:
1. 쿠폰번호 입력
2. 조회(L0) 결과 표시: 상품명, 유효기간, 사용 여부
3. 사용자가 "사용하기"를 누르거나 닫기

> 이 스토리는 US-1/US-2의 3단계에 해당합니다. check API가 별도로 필요한지, use API에 통합할지는 클라이언트 플로우에 따라 결정됩니다.

---

### US-4. 관리자 — 상품 관리

> 운영자가 카카오 선물하기에 등록할 상품(하트충전권/스킬교환권)을 AdminJS에서 관리한다.

**흐름 (하트 충전권)**:
1. 쿠프마케팅 측에서 신규 상품코드를 할당받는다
2. 운영자가 AdminJS에서 상품을 등록한다 (상품코드, 상품명, 유형=heart, 금액, 하트수량)
3. 상품을 활성화하면 사용자가 해당 상품권을 사용할 수 있다
4. 운영 중단 시 상품을 비활성화한다 (기존 사용 이력은 유지)

**흐름 (스킬 교환권)**:
1. 쿠프마케팅 측에서 신규 상품코드를 할당받는다
2. 운영자가 AdminJS에서 상품을 등록한다 (상품코드, 상품명, 유형=skill, 금액, 대상 스킬)
3. **시스템이 자동으로 "스킬 이용권" (100% 할인 쿠폰)을 생성한다** → US-4A 참조
4. 상품을 활성화하면 사용자가 해당 상품권을 사용할 수 있다
5. 운영 중단 시 상품을 비활성화한다 → 연결된 스킬 이용권도 함께 발급 중단된다

---

### US-4A. 관리자 — 스킬 이용권 자동 생성 및 관리

> 운영자가 스킬 교환권 상품을 등록하면 시스템이 해당 스킬의 100% 할인 쿠폰("스킬 이용권")을 자동 생성하고, 상품의 수정/비활성화/삭제 시 스킬 이용권도 연동하여 관리한다.

**사전 조건**:
- 운영자가 super 권한을 보유
- 대상 스킬(fixedMenuSeq)이 존재

#### 생성 흐름

1. 운영자가 상품 유형 `skill`을 선택하고 대상 스킬(fixedMenuSeq)을 입력한다
2. 상품 저장(생성) 시 시스템이 자동으로:
   - `CouponSpec` 생성: 이름 "**[상품명] 이용권**", 할인 타입 PERCENTAGE 100%, 플랫폼 ALL
   - `CouponCondition` 생성: 해당 스킬(skillSeqs=[fixedMenuSeq])에만 적용
   - 생성된 `CouponSpec.seq`를 상품의 `couponSpecSeq`에 자동 연결
3. 운영자에게 생성 완료 메시지 표시: "상품 등록 완료. 스킬 이용권(CouponSpec #{seq})이 자동 생성되었습니다."

#### 수정 흐름

1. 운영자가 스킬 교환권 상품을 수정한다
2. **상품명 변경** 시 → 연결된 CouponSpec.name도 "[변경된 상품명] 이용권"으로 자동 업데이트
3. **대상 스킬(fixedMenuSeq) 변경** 시 → 연결된 CouponCondition.skillSeqs도 자동 업데이트
4. 운영자에게 수정 완료 메시지 표시: "상품 수정 완료. 스킬 이용권도 함께 업데이트되었습니다."

#### 비활성화(이용중지) 흐름

1. 운영자가 상품의 `isActive`를 `false`로 변경한다
2. 시스템이 연결된 CouponSpec의 `issueEndDate`를 현재 시각으로 설정 (신규 발급 중단)
3. **이미 발급된 쿠폰은 유효 기간 내 사용 가능** (기발급 쿠폰 회수 없음)
4. 운영자에게 안내: "상품이 비활성화되었습니다. 신규 스킬 이용권 발급이 중단됩니다. 기발급 쿠폰은 유효 기간까지 사용 가능합니다."

#### 재활성화 흐름

1. 운영자가 상품의 `isActive`를 `true`로 변경한다
2. 시스템이 연결된 CouponSpec의 `issueEndDate`를 NULL로 복원 (발급 재개)
3. 운영자에게 안내: "상품이 활성화되었습니다. 스킬 이용권 발급이 재개됩니다."

#### 삭제(논리 삭제) 흐름

1. 운영자가 상품 삭제를 요청한다
2. 시스템이 상품의 `isActive`를 `false`로 변경 (물리 삭제 없음)
3. 연결된 CouponSpec의 `issueEndDate`를 현재 시각으로 설정
4. 비활성화와 동일한 처리

**예외 흐름**:
- 생성 시 CouponSpec 생성 실패 → 상품 등록도 롤백 (트랜잭션)
- 수정 시 CouponSpec 업데이트 실패 → 상품 수정도 롤백 (트랜잭션)
- 기발급 쿠폰이 있는 상품의 대상 스킬 변경 시 → 경고: "이미 발급된 이용권 N건이 있습니다. 기발급 이용권의 대상 스킬도 함께 변경됩니다."

---

### US-4B. 관리자 — 스킬 이용권 직접 조회/관리

> 운영자가 스킬 이용권(100% 할인 쿠폰)의 발급 현황을 조회하고, 필요 시 개별 이용권을 관리한다.

**흐름**:
1. 운영자가 CoupcMarketingProduct 상세 화면에서 연결된 `couponSpecSeq` 링크를 클릭한다
2. 기존 CouponSpec 어드민 상세 화면으로 이동하여 발급 현황(issuedQuantity, usedCount) 확인
3. CouponSpec의 conditions를 통해 대상 스킬 확인
4. 필요 시 CouponSpec의 발급 조건 세부 수정 (usableDays, usableCountLimitPerUser 등)

> **참고**: 스킬 이용권의 핵심 필드(name, discountType, discountValue, skillSeqs)는 상품 관리(US-4A)를 통해 자동 관리되므로 직접 수정하지 않는 것을 권장. 발급 정책(기간, 횟수 제한 등)만 직접 수정.

---

### US-5. 관리자 — 문제 해결 (사용 취소)

> 운영자가 사용자 문의를 받아 상품권 사용 이력을 조회하고, 필요 시 취소 처리한다.

**흐름**:
1. 사용자가 CS 문의 (예: "하트가 충전되지 않았어요")
2. 운영자가 AdminJS에서 쿠폰번호 또는 userSeq로 사용 이력을 검색한다
3. API 호출 로그를 확인하여 어느 단계에서 문제가 발생했는지 파악한다
4. 필요 시 "취소" 액션으로 사용취소(L2) 처리한다
5. 사용 이력 상태가 `canceled`로 변경되고, 쿠폰이 원복된다

**취소 정책 (쿠프마케팅 확인 완료)**:
- **사용 후 취소 불가**: 콘텐츠 제공형 상품이므로 정상 사용 후 사용자 요청에 의한 취소는 불가
- L2 취소는 **내부 오류 시에만** 사용 (L1 성공 후 하트 충전/쿠폰 발급 실패)
- 고객 클레임: 쿠프마케팅에서 쿠폰번호 전달 → 운영자가 정상 승인여부 확인하여 회신 (고객정보 마스킹)

---

### US-6. 관리자 — 정산 검증

> 운영자가 월말에 쿠프마케팅 정산서와 헬로우봇 내부 데이터를 대조하여 정산 금액을 검증한다.

**흐름**:
1. 운영자가 정산 통계 페이지에서 해당 월을 선택한다
2. 요약 통계를 확인한다: 총 사용 건수, 총 취소 건수, 순 정산 금액
3. 쿠프마케팅 정산서의 수치와 비교한다
4. 불일치 시 상품별/일별 내역을 드릴다운하여 원인을 파악한다

---

## 2. 구체화 필요 사항

스토리를 실제로 구현하려면 아래 항목들이 확정되어야 합니다.

### 2-1. 클라이언트 플로우 — ✅ 와이어프레임 v3에서 확정

> 상세: [wireframe-v3.html](./wireframe-v3.html), [screen-plan.md](./screen-plan.md)

| # | 항목 | 상태 | 결정 |
|---|------|------|------|
| C1 | **쿠폰 입력 화면의 진입점** | ✅ 확정 | 기존 쿠폰 탭 재사용 (프로필 → 쿠폰) + 카카오 딥링크. 별도 화면 신규 개발 없음. |
| C2 | **조회(check)와 사용(use) 분리** | ✅ 확정 | check → 확인 팝업 → use 2단계. 서버가 코드 형식을 판별하여 기존 쿠폰/카카오 상품권 분기. |
| C3 | **사용 완료 후 화면 전환** | ✅ 확정 | 하트: 충전 완료 팝업 → 충전 확인하기 → 내 하트 페이지. 스킬: 팝업 닫힘 → 토스트 + 쿠폰 리스트에 이용권 카드 → 스킬 상세 → 챗봇 채팅방. |
| C4 | **에러 메시지 UX** | ✅ 확정 | 에러코드별 사용자 메시지 정의 완료 (screen-plan.md S5 참조) |
| C5 | **쿠폰번호 입력 방식** | ✅ 확정 | 기존 입력란 그대로 사용 (붙여넣기 지원). 화면 유형: 네이티브. |

### 2-2. 스킬 교환 상세 설계 — ✅ 확정

**확정된 방식**: 100% 할인 쿠폰 발급 → 기존 스킬 구매 플로우로 0하트 구매

| # | 항목 | 상태 | 결정 내용 |
|---|------|------|----------|
| S1 | **스킬 제공 = 구체적으로 무엇인가** | ✅ 확정 | L1 성공 시 해당 스킬의 100% 할인 쿠폰(CouponSpec PERCENTAGE 100%)을 발급. 유저가 기존 스킬 구매 플로우에서 이 쿠폰을 적용하여 0하트로 구매. PremiumSkillLog는 기존 구매 플로우에서 자동 생성됨. |
| S2 | **기프티엘의 useCoupon 호출 위치** | ✅ 불필요 | 기프티엘 패턴을 따르지 않고 쿠폰 발급 방식으로 변경되어 해당 검토 불필요 |
| S3 | **이미 스킬을 보유한 사용자가 교환권을 사용하면?** | ✅ 확정 | 쿠폰 발급은 정상 진행 (L1 사용 처리 완료). 스킬 구매 시점에 기존 중복 구매 체크 로직이 적용됨. 쿠폰은 보관되므로 유저가 원할 때 사용 가능. |

### 2-3. 취소/원복 정책 — ✅ 확정 (쿠프마케팅 답변 기반)

> **기본 정책**: 콘텐츠 제공형 상품이므로 **사용 후 취소 불가**. L2 취소는 내부 오류 시에만 사용.

| # | 항목 | 상태 | 결정 |
|---|------|------|------|
| R1 | **취소 시 하트 차감 여부** | ✅ 확정 | 사용 후 취소 불가 정책이므로 **정상 사용 후 하트 차감은 하지 않음**. L1 성공 후 내부 충전 실패 시에만 L2 취소 → 이 경우 하트가 충전되지 않았으므로 차감 불필요. |
| R2 | **취소 시 스킬 이용권 회수** | ✅ 확정 | 사용 후 취소 불가 정책. L1 성공 후 쿠폰 발급 실패 시에만 L2 취소 → 이 경우 쿠폰이 발급되지 않았으므로 회수 불필요. |
| R3 | **자동 취소 vs 수동 취소** | ✅ 확정 | L1 성공 후 내부 처리(하트 충전/쿠폰 발급) **실패 시 자동으로 L2 호출**. 고객 클레임은 쿠프마케팅에서 쿠폰번호 전달 → 정상 승인여부 확인하여 회신 (고객정보 마스킹). |
