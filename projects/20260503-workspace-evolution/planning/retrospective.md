## 3. 반복 비용 패턴 (제거할 것)

### 3-1. 도메인 지식의 재발견

**관찰**:
- coop-integration 에서 `/dev-server` 가 학습한 쿠폰/하트 도메인 지식 (예: `chargeHeart` 와 usage 트랜잭션 분리, `usableDays` 정책, 쿠프마케팅 거래번호 길이 제약) 은 ISS-001 같은 결정 로그에만 남아 있음
- 다음 결제 관련 프로젝트가 시작되면 같은 코드를 다시 grep 해야 도메인을 재구성
- `dev-data` 는 `data-infra-documentation` 시리즈로 카탈로그 SSOT 가 형성되어 이 문제가 어느 정도 해결되어 있음 — 다른 리포에 같은 모델 적용 필요

**개선**:
- 각 리포에 `docs/domain/` 신설하여 "코드/이전 프로젝트 결정에서 추출한 도메인 자산" 영속 보관 (자세한 설계: [domain-knowledge-design.md](./domain-knowledge-design.md))

### 3-2. 반복 작업 절차의 비형식화

**관찰**:
- "TypeORM 마이그레이션 추가", "AdminJS 리소스 추가", "BQ 마트 추가", "신규 Firebase 이벤트 추가" 등은 매 프로젝트 반복
- `event-spec.md` 는 5단계 검증 절차를 포함 → 여기까진 형식화됨
- 그러나 서버/웹/iOS/Android 의 반복 절차는 매번 에이전트가 처음부터 추론

**개선**:
- 반복 절차를 리포 `docs/patterns/` 또는 스킬로 형식화
- `/dev-*` 커맨드 본문에 "이 작업이 patterns/ 에 있으면 먼저 참조" 절차 추가

### 3-3. 컨텍스트 진입 비용

**관찰**:
- 신규 세션마다 에이전트가 워크스페이스 → 프로젝트 → 리포 → 코드 순으로 광범위 탐색
- 특히 처음 보는 도메인은 grep/glob 다회 호출로 컨텍스트 부풀림
- 메인 컨텍스트가 무거워지면 후속 작업의 정확도 저하

**개선**:
- Explore 서브에이전트로 광범위 탐색 위임
- 도메인 진입점(`docs/domain/index.md`)을 메모리 인덱스로 격상

### 3-4. 운영 규칙 ↔ 실제 사용의 간극

**관찰 사례**:
- CLAUDE.md 는 `/dev-*` 가 `planning/` 을 직접 참조하지 못하게 명시 — 실제 거의 지켜지나 사람이 의식적으로 점검할 때만 발견됨
- "과업 완료 시 tasks.md 체크" 는 잊혀지는 일이 적지만, "계약 문서 변경 시 Changelog 기록" 은 누락 종종 발생
- `/workspace` 점검이 catch-up 역할을 했으나 사후 보정

**개선**:
- 결정론적 자동화로 강제 (hooks)
- 예: 계약 문서(`architecture.md` / `api-spec.md`) edit 후 Changelog 미수정 감지 시 경고

### 3-5. 이전 프로젝트 결정의 비참조

**관찰**:
- coop-integration 의 ISS-001~ISS-049 결정은 다음 결제 프로젝트 진행 시 자동 참조되지 않음
- `dbt-migration-prep` 의 17 finding 은 후속 dbt-migration 프로젝트가 직접 인풋으로 받음 → 명시적 인계가 있을 때만 활용됨

**개선**:
- 의사결정 카드(ADR) 를 리포 `docs/decisions/` 에 영속화
- 도메인 진입점이 관련 ADR 을 인덱싱
- 새 프로젝트의 `/analyze` 가 자동으로 관련 ADR 을 참조 후보로 제시

### 3-6. 메모리 시스템의 미활용

**관찰**:
- 7개 프로젝트를 거치며 메모리 항목은 5건에서 거의 변동 없음
- 누적되어야 할 도메인 진입점, 자주 발생하는 의사결정 단축 메모가 없음

**개선**:
- 프로젝트 종료 절차에 "메모리 갱신 필요 여부 점검" 추가
- 단, 메모리는 인덱스/단축이고 본문은 docs/

## 4. 운영 규칙 ↔ 실제 사용의 간극 (현실화할 것)

### 4-1. 종료 절차의 일관성

**관찰**:
- `/workspace 종료` 절차는 잘 정형화되어 있음 (tasks 정리 / 이슈 정리 / PR 수집 / SSOT 승격)
- 단, "메모리 갱신" / "도메인 자산 갱신" 단계가 빠져 있음

**현실화**:
- 종료 절차에 "도메인 자산(`docs/domain/`, `docs/patterns/`, `docs/decisions/`) 갱신 필요 여부" 점검 단계 추가

### 4-2. 리포 status.md 결정 로그 활용

**관찰**:
- 결정 로그는 본 프로젝트의 의사결정 기록으로는 잘 동작
- 단, **다음 프로젝트가 자동 참조하지 않음** — 단지 그 프로젝트 디렉토리에 묻혀 있음

**현실화**:
- 다음 프로젝트가 참조해야 할 결정은 ADR 로 승격 (영속화)
- 리포 status 결정 로그는 이번 프로젝트 한정 (운영 그대로)

### 4-3. tasks.md 분류의 안정성

**관찰**:
- 파트별(서버/iOS/...) 구분은 잘 동작
- "기획" 섹션이 점점 비대해지는 경향 (planning/ 이 잘 활용된 프로젝트 한정)
